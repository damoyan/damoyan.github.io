多个手势共用的情况

当有多个手势添加到一个View上的时候, 默认的是没有识别顺序的, 也就意味着每次touch传递给gesture recognizer的顺序都是随机的. 官方文档指出我们可以通过三种方式来修改默认的行为:

  • 指定一个GestureRecognizerA在另一个GestureRecognizerB之前解析touch事件
  • 允许两个GestureRecognizer同时运作
  • 阻止一个GestureRecognizer解析touch事件

指定两个GestureRecognizer的解析顺序

UIGestureRecognizer提供了requireGestureRecognizerToFail:方法可以做到.

这个方法的意思是: 只有当gestureB无法解析touch事件(即状态进入Failed状态)的时候, gestureA才开始解析touch事件, 此时gestureA处于Possible状态. 换句话说, 如果gestureB解析了touch事件并且进入了Began状态, 那么gestureA会直接Fail.

阻止GestureRecognizer解析事件

通过实现UIGestureRecognizerDelegate协议提供的两种方法都可以做到此要求: gestureRecognizer:shouldReceiveTouch:和gestureRecognizerShouldBegin:.

每当系统收到一个新的touch的时候gestureRecognizer:shouldReceiveTouch:就会被调用. 实现该方法并返回NO, 那么该GestureRecognizer就不会收到关于这个touch的信息.这个方法是在touchesBegan:withEvent:之前调用的.

gestureRecognizerShouldBegin:是在GestureRecognizer将要从Possible状态转出的时候调用的, 如果返回NO, 那么GestureRecognizer会立即转换到Failed状态, 那么其他的事件处理者就可以接入了.

两个GestureRecognizer同时运作

实现UIGestureRecognizerDelegate协议的gestureRecognizer:shouldRecognizeSimultaneouslyWithGestureRecognizer:方法, 并且返回YES.

注意:

如果你有两个UIGestureRecognizer,假如是A和B, 那么你需要在A或者B上实现这个方法并且返回YES就可以允许两个同时运作. 但是如果想要阻止他们同时运作, 那么需要保证这两个UIGestureRecognizer的这个方法都返回NO.

在两个GestureRecognizer指定单向的关系(Specifying a One-Way RelationShip Between Two Gesture Recognizer)

你可以实现一个子类, 继承一个手势, 然后实现`GestureRecognizerSubclass.h里面的canPreventGestureRecognizer:和canBePreventedByGestureRecognizer:来实现单向的关系. 例如: 默认的情况下, pinch gesture和rotation gesture之间是相互阻止的, 因为它们不可能被同时识别.

类的构造函数

类里面的所有存储属性—包括所有继承自父类的属性—都必须在构造函数中设置初始值.

Swift为类提供了两种类型的类构造函数: 指定构造函数(Designated)和便利构造函数(Convenience).

指定构造函数是类中最主要的构造函数. 指定构造函数初始化类中提供的所有属性, 并根据父类链往上调用父类的构造器. 每个类都必须至少拥有一个指定构造函数. 在某些情况下, 类通过继承父类中的指定构造函数满足这个条件.

便利构造函数是类中比较次要的, 辅助型构造函数. 你可以定义便利构造函数来调用同一个类中的指定构造函数, 并为其参数提供默认值. 你也可以定义便利构造函数来创建一个特殊用途的实例.

构造函数的调用规则:

  1. 指定构造函数必须调用其直接父类的指定构造函数
  2. 便利构造函数必须调用同一类中定义的其他构造函数
  3. 便利构造函数必须以调用一个指定构造函数结束

一个更方便记忆的方法:

  • 指定构造函数必须总是向上代理
  • 便利构造函数必须总是横向代理

类的构造过程 - Two-Phase Initialization

构造过程分为两步:

  1. 初始化所有在此类本身定义的存储属性.
  2. 在所有存储属性初始化完成之后, 在类被使用之前, 还有一次机会来自定义它的存储属性

为了保证两段构造过程不出问题, Swift会在编译阶段进行4项检查:

  1. 在指定构造函数里面, 必须先对所有在此类里面定义的存储属性进行初始化之后, 才能调用父类的构造函数
  2. 指定构造函数里面, 必须先调用父类的构造函数之后, 才能对从父类继承过来的属性进行赋值. 否则的话, 在子类构造函数里面赋的值会被父类的值覆盖掉.
  3. 一个Convenience构造函数必须先调用Designated构造函数, 然后才能给属性赋值(包括当前类的或者父类的), 否则, 在这个构造函数里面赋的值就会被覆盖掉.
  4. 在构造过程的第一阶段完成之前, 构造函数不能调用任何实例方法, 不能读取任何实例属性, 不能使用self.

可选类型(Optional)

可选(Optional)类型, 表示”那儿有个值, 并且它等于x”或者”那儿没有值”, 它有点像nil, 但是可以用在任何类型上, 比nil更加安全也更有表现力.

注意:

C 和 Objective-C 中并没有可选类型这个概念。最接近的是 Objective-C 中的一个特性，一个方法要不返回一个对象要不返回nil，nil表示“缺少一个合法的对象”。然而，这只对对象起作用——对于结构体，基本的 C 类型或者枚举类型不起作用。对于这些类型，Objective-C 方法一般会返回一个特殊值（比如NSNotFound）来暗示值缺失。这种方法假设方法的调用者知道并记得对特殊值进行判断。然而，Swift 的可选类型可以让你暗示任意类型的值缺失，并不需要一个特殊值。

nil

注意

Swift 的nil和 Objective-C 中的nil并不一样。在 Objective-C 中，nil是一个指向不存在对象的指针。在 Swift 中，nil不是指针——它是一个确定的值，用来表示值缺失。任何类型的可选状态都可以被设置为nil，不只是对象类型。

隐式解析可选类型

有时候在程序架构中，第一次被赋值之后，可以确定一个可选类型总会有值。在这种情况下，每次都要判断和解析可选值是非常低效的，因为可以确定它总会有值。

这种类型的可选状态被定义为隐式解析可选类型（implicitly unwrapped optionals）。把想要用作可选的类型的后面的问号（String?）改成感叹号（String!）来声明一个隐式解析可选类型。

最近看了@孙源的一个介绍RunLoop的视频，学到一些东西，记录在这。

RunLoop的创建

需要注意的是, 这行代码需要在runloop所在的线程调用. 当所在线程的runloop已经存在时, 此方法会直接返回该runloop, 如果不存在, 此方法会先创建runloop.

RunLoop的运行

在运行runloop之前, 需要给它添加事件源, runloop的事件源分为三种类型:

  • Input Source (performSelector, 监听port)
  • Timer
  • Observer (只能通过CoreFoundation的API添加)

如果没有事件源的话, runloop会立即退出. Input Source从NSRunLoop的头文件看, 主要用的就是performSelector:和addPort:forMode:两种方式. 实际上还有其他的方式, 甚至可以自定义, 但是我还没研究过.

添加Timer

Timer有两种初始化的方式:

  1. timeWithTimeInterval:…
  2. scheduledTimerWithTimeInterval:…

1会创建一个Timer, 但是不会触发, 需要手动的添加到runloop里面去调度. 2创建一个Timer并且把它放到runloop的NSDefaultRunLoopMode里面去调度.

1比2更灵活, 当我们滑动UIScrollView的时候, runloop是处在NSEventTrackingRunLoopMode, 这个时候如果通过2添加的Timer是不会被触发的, 因为Mode不对. 如果我们想在滑动view的时候Timer不被影响的话, 可以用1的方式.

NSRunLoopCommonModes不是一个具体的mode, 它是一个mode的集合(NSDefaultRunLoopMode和NSEventTrackingRunLoopMode都属于这个集合), 如果添加timer的时候用这个mode, 那么这个timer就会被添加到集合里面的每一个mode. 这样在滑动的时候timer也能被正常的触发了.

注意!

在使用runMode:beforeDate:方法的时候, mode不能被指定为NSRunLoopCommonModes! 因为它不是一个具体的mode, 如果指定了, runloop会立即退出.
